# Deploy notes — 2026-07-05 (0.5.43 → 0.5.53)

## Do these on deploy

1. **Redeploy** the server (carries all eleven commits, 0.5.43 through 0.5.53).
2. **Instagram video:** set `GOHORT_YTDLP_COOKIES` in gohort's environment to a
   `cookies.txt` exported from a browser logged into Instagram. Use a **throwaway
   account** (automated downloading can flag your main one). Without this,
   Instagram reels still won't download; it is their auth policy, not a code
   issue.
3. **Recommended:** update yt-dlp, it is stale at `2026.03.17`. Run
   `sudo yt-dlp -U` (it is a self-updating standalone binary, not pip). Avoid
   `pip install` — it ties yt-dlp to a Python that can be removed out from under it.
4. **Verify after deploy:** if video *evaluation* is still off, confirm the
   llama.cpp worker was launched with the vision projector (`--mmproj`). Frames
   now reach it, but a text-only server would silently drop them.

## New environment variables

- `GOHORT_YTDLP_COOKIES` — path to a `cookies.txt` for login-gated video sites
  (Instagram, and any other site yt-dlp needs a session for).
- `GOHORT_YTDLP_COOKIES_FROM_BROWSER` — a browser name (e.g. `firefox`), an
  alternative to the cookies file for desktop hosts.

## Behavior changes to know

- **Eval endpoint default flipped (0.5.44).** `POST /api/agents/{id}/eval` is now
  **stub-by-default**: no real tools fire. Add `?live=1` to run real
  non-consequential tools, or `?live=all` to also allow approval-gated ones.
  Admin-only, but any automation hitting it expecting live tool runs now needs
  the flag.
- **Agents are more honest (0.5.46, 0.5.52).** They will no longer dress a date
  with an invented holiday, and will not describe an image, video, or link they
  could not actually open or download. Expect "I couldn't pull that reel" instead
  of a fabricated description.

## What shipped

### Video pipeline (the Instagram thread)
- **0.5.49** yt-dlp cookie support plus a clear "needs login" error.
- **0.5.51** fixed `view_video`: sampled frames now actually reach the model. They
  were being extracted and dropped, which is what caused the hallucinated
  descriptions of reels the agent never saw.
- **0.5.52** media-honesty grounding guard for when a download or view fails.

### Servitor repo refresh (Phases A/B/C)
- **0.5.47** refresh is now a streamed, cancelable session (spinner + Cancel),
  and Cancel aborts the git clone mid-flight.
- **0.5.48** on re-clone it validates its knowledge docs against the new code and
  auto-corrects stale ones.
- **0.5.53** and reconciles discrete facts: corrects changed values, retires facts
  whose subject is gone from the code. Adds the graph-attr retire primitive the
  fact-store split had left missing.

### Operability
- **0.5.50** dependency-health surface: the admin "System Dependencies" panel now
  shows external-tool versions and flags stale ones (yt-dlp especially); a boot
  log line warns on anything missing or stale. This is what makes the next silent
  breakage visible instead of showing up only when a feature quietly fails.

### Quality / plumbing
- **0.5.45** Guides and techwriter servitor pulls are now query-focused (a
  relevant slice) instead of dumping the whole appliance's knowledge.
- **0.5.43** trimmed a redundant prompt block in `roundShapePreamble` (no behavior
  change; the delegate-first / answer-after-tools guidance it restated is already
  carried by the always-on framework blocks).

## Post-deploy smoke checks

- Text a real Instagram reel to the phantom channel: it should download, describe
  the actual content, and if blocked, say so honestly rather than guessing.
- Hit **Refresh Repo** on a repo appliance: expect a spinner, live status lines, a
  Cancel button, and a "validated / reconciled memory" summary.
- Admin → **System Dependencies**: yt-dlp should now appear with a version and,
  until you update it, a "Stale, update" badge.
